Estás instalando Coasts en este proyecto. Coast (host contenedorizado) es una herramienta CLI que ejecuta múltiples entornos de desarrollo aislados en una sola máquina usando contenedores Docker-in-Docker. Cada entorno obtiene sus propios puertos, volúmenes y runtime — ideal para flujos de trabajo paralelos con worktrees.

Tu trabajo: analizar este proyecto y generar un Coastfile (un archivo TOML llamado "Coastfile" en la raíz del proyecto).

=== DOCUMENTACIÓN ===

Coast tiene documentación integrada accesible desde la CLI. Úsala para entender el esquema completo de Coastfile, las estrategias de volúmenes, el comportamiento de assign y otras opciones de configuración antes de generar un Coastfile.

Explora el árbol de documentación:

  coast docs

Esto imprime el árbol completo de documentación. Empieza leyendo los archivos README — proporcionan índices para ayudarte a encontrar la documentación adecuada para cada tema:

  coast docs --path README.md
  coast docs --path coastfiles/README.md
  coast docs --path concepts_and_terminology/README.md

Lee un documento específico:

  coast docs --path coastfiles/PROJECT.md
  coast docs --path coastfiles/VOLUMES.md

Busca en la documentación (búsqueda semántica — describe lo que buscas en lenguaje natural):

  coast search-docs "how do volume strategies work"

  coast search-docs "secret injection from environment variables"

Usa la documentación para tomar decisiones informadas sobre la configuración del Coastfile de este proyecto. La sección coastfiles/ cubre cada directiva de Coastfile en detalle.

=== ESQUEMA DE COASTFILE (referencia rápida) ===

[coast] — Requerido. Metadatos del proyecto.

  name        (string, required)  Identificador del proyecto usado en el nombrado de contenedores/volúmenes.
  compose     (string, optional)  Ruta a docker-compose.yml relativa al Coastfile.
  runtime     (string, optional)  "dind" (predeterminado), "sysbox" o "podman".
  root        (string, optional)  Anulación de la raíz del proyecto (relativa o absoluta).

[coast.setup] — Opcional. Personaliza el contenedor DinD en sí.

  packages    (array of strings)  Paquetes de Alpine para instalar (p. ej. ["nodejs", "npm", "git"]).
  run         (array of strings)  Comandos arbitrarios para ejecutar durante la configuración.

[ports] — Requerido (al menos uno). Mapa de nombre lógico a número de puerto.
  Estos puertos se reenvían al host cuando el coast se hace checkout.

  Ejemplo:
  [ports]
  web = 3000
  api = 8080
  postgres = 5432

[volumes.*] — Opcional. Configuración por volumen.

  strategy        "isolated" (predeterminado) o "shared"
  service         Nombre del servicio de Compose que posee este volumen.
  mount           Ruta de montaje dentro del contenedor del servicio.
  snapshot_source (solo isolated) Semilla desde un nombre de volumen existente.

  Ejemplo:
  [volumes.postgres_data]
  strategy = "isolated"
  service = "db"
  mount = "/var/lib/postgresql/data"

[secrets.*] — Opcional. Extracción e inyección de secretos.

  extractor   "file", "env", "command" o "macos-keychain"
  inject      "env:VAR_NAME" o "file:/path/in/container"
  ttl         Caducidad opcional (p. ej. "1h", "30m").

  Parámetros específicos del extractor:
    file:            path = "./path/to/secret"
    env:             var = "HOST_ENV_VAR"
    command:         run = "echo secret-value"
    macos-keychain:  item = "keychain-item-name"

  Ejemplo:
  [secrets.db_password]
  extractor = "env"
  var = "DB_PASSWORD"
  inject = "env:DATABASE_PASSWORD"

[inject] — Opcional. Inyección de archivos/variables de entorno del host que no son secretos.

  env     Array de nombres de variables de entorno del host para reenviar.
  files   Array de rutas de archivos del host para montar.

  Ejemplo:
  [inject]
  env = ["NODE_ENV", "DEBUG"]
  files = ["~/.ssh/id_ed25519", "~/.gitconfig"]

[shared_services.*] — Opcional. Servicios en el daemon Docker del host compartidos entre instancias.

  image           Imagen de Docker.
  ports           Array de números de puerto.
  volumes         Array de montajes de volumen.
  env             Tabla inline de variables de entorno.
  auto_create_db  (bool) Crear automáticamente una base de datos por instancia.
  inject          Inyectar la cadena de conexión en los contenedores coast.

  Ejemplo:
  [shared_services.postgres]
  image = "postgres:16-alpine"
  ports = [5432]
  volumes = ["postgres_data:/var/lib/postgresql/data"]
  env = { POSTGRES_USER = "dev", POSTGRES_PASSWORD = "dev", POSTGRES_DB = "app" }
  auto_create_db = true
  inject = "env:DATABASE_URL"

[assign] — Opcional. Controla lo que sucede al cambiar de rama (coast assign).

  default                 "none", "restart" o "rebuild"
  [assign.services]       Anulaciones por servicio.
  [assign.rebuild_triggers]  Globs de archivos por servicio que disparan rebuild.

  Ejemplo:
  [assign]
  default = "none"
  [assign.services]
  api = "restart"
  worker = "rebuild"
  [assign.rebuild_triggers]
  worker = ["Dockerfile", "package.json"]

[services.*] — Opcional. Servicios de proceso simple (no se necesita docker-compose).

  command   Comando de shell a ejecutar.
  port      Número de puerto.
  restart   "on-failure" o "always".

  Ejemplo:
  [services.web]
  command = "node server.js"
  port = 3000
  restart = "on-failure"

=== EJEMPLO: Mínimo (sin compose) ===

[coast]
name = "my-app"
runtime = "dind"

[coast.setup]
packages = ["nodejs", "npm"]

[ports]
app = 3000

=== EJEMPLO: Con docker-compose ===

[coast]
name = "my-app"
compose = "./docker-compose.yml"
runtime = "dind"

[ports]
app = 3000
postgres = 5432
redis = 6379

[volumes.postgres_data]
strategy = "shared"
service = "db"
mount = "/var/lib/postgresql/data"

[volumes.redis_data]
strategy = "isolated"
service = "cache"
mount = "/data"

[assign]
default = "none"

[assign.services]
app = "rebuild"

=== EJEMPLO: Con secretos ===

[coast]
name = "my-app"
compose = "./docker-compose.yml"
runtime = "dind"

[ports]
app = 3000

[secrets.api_key]
extractor = "env"
var = "API_KEY"
inject = "env:API_KEY"

[secrets.ssh_key]
extractor = "file"
path = "~/.ssh/id_ed25519"
inject = "file:/run/secrets/ssh_key"

=== PRINCIPALES COMPENSACIONES PARA DISCUTIR CON EL USUARIO ===

Antes de generar el Coastfile, pregunta al usuario sobre cualquier elección de configuración ambigua. Estas son las principales:

Estrategia de base de datos e infraestructura — hay tres opciones para servicios como postgres y redis:
  - Volúmenes aislados (predeterminado): cada instancia de Coast obtiene su propia copia de los datos dentro de su contenedor DinD. Las instancias no pueden interferir entre sí. Mejor cuando quieres estado de base de datos por rama.
  - Volúmenes compartidos: todas las instancias leen y escriben el mismo volumen dentro de sus contenedores DinD. Ahorra espacio en disco, pero escrituras concurrentes de múltiples instancias pueden corromper datos.
  - Servicios compartidos: ejecuta la base de datos en el daemon Docker del host en lugar de dentro de cada Coast. Todas las instancias se conectan a un único servidor compartido. Usa la menor memoria, soporta auto_create_db para bases de datos por instancia en un solo postgres, y los datos sobreviven a la eliminación de instancias. Mejor para equipos grandes o máquinas con memoria limitada.
  - Si el proyecto tiene una base de datos, pregunta al usuario qué enfoque quiere. Explica las compensaciones — isolated es lo más seguro, shared services es lo más eficiente en memoria.

Estrategia de assign — qué sucede al cambiar un Coast entre worktrees:
  - "none": no hacer nada (para servicios como postgres/redis que no cambian entre ramas).
  - "restart": reiniciar el contenedor (para servicios interpretados que solo necesitan reiniciar el proceso).
  - "rebuild": reconstruir la imagen Docker y reiniciar (para servicios donde el cambio de rama afecta el Dockerfile o dependencias de build).
  - Si el proyecto tiene múltiples servicios, pregunta cuáles necesitan rebuild vs restart al cambiar de rama.

=== INSTRUCCIONES ===

1. Mira la estructura de este proyecto. Si hay un docker-compose.yml, léelo para identificar servicios, puertos y volúmenes.
2. Lee la documentación relevante de Coast (usa `coast docs` y `coast search-docs`) para entender estrategias de volúmenes, comportamiento de assign y cualquier opción de configuración que aplique al stack de este proyecto.
3. Pregunta al usuario sobre cualquier elección de configuración ambigua (ver compensaciones arriba). No adivines — explica las opciones y deja que decidan.
4. Genera un Coastfile en la raíz del proyecto basado en el análisis del proyecto y la entrada del usuario.
5. Si el proyecto no tiene docker-compose.yml, usa [services.*] para definiciones de procesos simples o [coast.setup] para instalar dependencias.
6. Ejecuta `coast build`. Si falla, revisa el error y consulta la documentación (`coast search-docs "<error description>"`) para solucionarlo.
7. Ejecuta `coast run dev-1`. Si falla, revisa el error y consulta la documentación.
8. Ejecuta `coast ui` para abrir el panel de Coastguard (esto es para el usuario cuando hayas terminado).
