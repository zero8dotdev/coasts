# Bare Services

Если вы можете контейнеризировать свой проект — сделайте это. Bare services существуют для проектов, которые ещё не контейнеризированы и где добавление `Dockerfile` и `docker-compose.yml` в краткосрочной перспективе непрактично. Это промежуточный шаг, а не конечная цель.

Вместо того чтобы `docker-compose.yml` оркестрировал контейнеризованные сервисы, bare services позволяют вам определить shell-команды в вашем Coastfile, а Coast запускает их как обычные процессы с лёгким супервизором внутри контейнера Coast.

## Why Containerize Instead

Сервисы [Docker Compose](RUNTIMES_AND_SERVICES.md) дают вам:

- Воспроизводимые сборки через Dockerfile
- Health check’и, которых Coast может дождаться при старте
- Изоляцию процессов между сервисами
- Управление томами и сетью, которое берёт на себя Docker
- Переносимое определение, которое работает в CI, staging и production

Bare services не дают ничего из этого. Ваши процессы делят одну и ту же файловую систему, восстановление после падений — это shell-цикл, а «работает у меня на машине» столь же вероятно внутри Coast, как и вне его. Если у вашего проекта уже есть `docker-compose.yml`, используйте его.

## When Bare Services Make Sense

- Вы внедряете Coast в проект, который никогда не был контейнеризирован, и хотите сразу начать получать ценность от изоляции worktree и управления портами
- Ваш проект — одно-процессный инструмент или CLI, где Dockerfile был бы излишним
- Вы хотите постепенно двигаться к контейнеризации — начать с bare services, а позже перейти на compose

## Configuration

Bare services задаются секциями `[services.<name>]` в вашем Coastfile. Coastfile **не может** одновременно определять и `compose`, и `[services]` — они взаимоисключающие.

```toml
[coast]
name = "my-app"
runtime = "dind"

[coast.setup]
packages = ["nodejs", "npm"]

[services.web]
install = "npm install"
command = "npx next dev --port 3000 --hostname 0.0.0.0"
port = 3000
restart = "on-failure"

[services.worker]
command = "node worker.js"
restart = "always"

[ports]
web = 3000
```

У каждого сервиса есть четыре поля:

| Field | Required | Description |
|---|---|---|
| `command` | yes | Команда shell для запуска (например, `"npm run dev"`) |
| `port` | no | Порт, на котором слушает сервис; используется для маппинга портов |
| `restart` | no | Политика перезапуска: `"no"` (по умолчанию), `"on-failure"` или `"always"` |
| `install` | no | Одна или несколько команд, которые выполняются перед стартом (например, `"npm install"` или `["npm install", "npm run build"]`) |

### Setup Packages

Поскольку bare services запускаются как обычные процессы, в контейнере Coast должны быть установлены нужные рантаймы. Используйте `[coast.setup]`, чтобы объявить системные пакеты:

```toml
[coast.setup]
packages = ["nodejs", "npm"]
```

Они устанавливаются до запуска любого сервиса. Без этого ваши команды `npm` или `node` будут падать внутри контейнера.

### Install Commands

Поле `install` выполняется перед стартом сервиса и снова при каждом [`coast assign`](ASSIGN.md) (переключение ветки). Сюда помещают установку зависимостей:

```toml
[services.api]
install = ["pip install -r requirements.txt", "python manage.py migrate"]
command = "python manage.py runserver 0.0.0.0:8000"
port = 8000
```

Команды install выполняются последовательно. Если какая-либо install-команда завершится с ошибкой, сервис не стартует.

### Restart Policies

- **`no`** — сервис запускается один раз. Если он завершился, он остаётся остановленным. Используйте это для одноразовых задач или сервисов, которые вы хотите управлять вручную.
- **`on-failure`** — перезапускает сервис, если он завершился с ненулевым кодом. Успешные завершения (код 0) не трогаются. Используется экспоненциальная задержка от 1 секунды до 30 секунд, и после 10 последовательных падений попытки прекращаются.
- **`always`** — перезапускает при любом завершении, включая успешное. Та же задержка, что и у `on-failure`. Используйте это для долгоживущих серверов, которые не должны останавливаться.

Если сервис проработал более 30 секунд перед падением, счётчик попыток и задержка сбрасываются — предполагается, что некоторое время он был здоров, и падение является новой проблемой.

## How It Works Under the Hood

```text
┌─── Coast: dev-1 ──────────────────────────────────────┐
│                                                       │
│   /coast-supervisor/                                  │
│   ├── web.sh          (runs command, tracks PID)      │
│   ├── worker.sh                                       │
│   ├── start-all.sh    (launches all services)         │
│   ├── stop-all.sh     (SIGTERM via PID files)         │
│   └── ps.sh           (checks PID liveness)           │
│                                                       │
│   /var/log/coast-services/                            │
│   ├── web.log                                         │
│   └── worker.log                                      │
│                                                       │
│   No inner Docker daemon images are used.             │
│   Processes run directly on the container OS.         │
└───────────────────────────────────────────────────────┘
```

Coast генерирует shell-обёртки для каждого сервиса и размещает их в `/coast-supervisor/` внутри DinD-контейнера. Каждая обёртка отслеживает свой PID, перенаправляет вывод в log-файл и реализует политику перезапуска как shell-цикл. Нет Docker Compose, нет внутренних Docker-образов и нет изоляции на уровне контейнеров между сервисами.

`coast ps` проверяет «живость» PID вместо опроса Docker, а `coast logs` читает log-файлы (tail) вместо вызова `docker compose logs`. Формат вывода логов соответствует формату compose `service | line`, поэтому UI Coastguard работает без изменений.

## Ports

Конфигурация портов работает ровно так же, как и в Coast на базе compose. Определите порты, которые слушают ваши сервисы, в `[ports]`:

```toml
[services.web]
command = "npm start"
port = 3000

[ports]
web = 3000
```

[Dynamic ports](PORTS.md) выделяются при `coast run`, а [`coast checkout`](CHECKOUT.md) как обычно меняет местами канонические порты. Единственное отличие — между сервисами нет Docker-сети: все они биндятся напрямую на loopback контейнера или на `0.0.0.0`.

## Branch Switching

Когда вы запускаете `coast assign` в Coast с bare services, происходит следующее:

1. Все запущенные сервисы останавливаются через SIGTERM
2. Worktree переключается на новую ветку
3. Команды install выполняются заново (например, `npm install` подхватывает зависимости новой ветки)
4. Все сервисы перезапускаются

Это эквивалентно тому, что происходит с compose — `docker compose down`, переключение ветки, пересборка, `docker compose up` — но с shell-процессами вместо контейнеров.

## Limitations

- **Нет health check’ов.** Coast не может ждать, пока bare service станет «здоровым», как он может с compose-сервисом, который определяет health check. Он запускает процесс и надеется на лучшее.
- **Нет изоляции между сервисами.** Все процессы делят одну и ту же файловую систему и пространство процессов внутри контейнера Coast. Неправильно работающий сервис может влиять на другие.
- **Нет кеширования сборки.** Сборки Docker Compose кешируются по слоям. `install`-команды bare services выполняются с нуля при каждом assign.
- **Восстановление после падений примитивное.** Политика перезапуска использует shell-цикл с экспоненциальной задержкой. Это не процессный супервизор вроде systemd или supervisord.
- **Нет `[omit]` или `[unset]` для сервисов.** Композиция типов Coastfile работает с compose-сервисами, но bare services не поддерживают исключение отдельных сервисов через типизированные Coastfile.

## Migrating to Compose

Когда вы будете готовы контейнеризировать, путь миграции прямолинеен:

1. Напишите `Dockerfile` для каждого сервиса
2. Создайте `docker-compose.yml`, который ссылается на них
3. Замените секции `[services.*]` в вашем Coastfile на поле `compose`, указывающее на ваш compose-файл
4. Удалите пакеты `[coast.setup]`, которые теперь будут устанавливаться вашими Dockerfile
5. Пересоберите с помощью [`coast build`](BUILDS.md)

Ваши настройки маппинга портов, [volumes](VOLUMES.md), [shared services](SHARED_SERVICES.md) и [secrets](SECRETS.md) без изменений переносятся. Меняется только то, как именно запускаются сами сервисы.
