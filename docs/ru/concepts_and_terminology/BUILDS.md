# Сборки

Думайте о сборке Coast как об образе Docker с дополнительной поддержкой. Сборка — это артефакт на основе директории, который упаковывает всё необходимое для создания экземпляров Coast: разрешённый [Coastfile](COASTFILE_TYPES.md), переписанный compose-файл, заранее скачанные OCI-тарболы образов и внедрённые файлы хоста. Это не образ Docker сам по себе, но он содержит образы Docker (в виде tarball’ов) плюс метаданные, которые нужны Coast, чтобы связать их воедино.

## Что делает `coast build`

Когда вы запускаете `coast build`, демон выполняет следующие шаги по порядку:

1. Разбирает и валидирует Coastfile.
2. Читает compose-файл и отфильтровывает пропущенные сервисы.
3. Извлекает [секреты](SECRETS.md) из настроенных экстракторов и сохраняет их в зашифрованном виде в хранилище ключей.
4. Собирает Docker-образы для compose-сервисов, у которых есть директивы `build:` (на хосте).
5. Скачивает Docker-образы для compose-сервисов, у которых есть директивы `image:`.
6. Кэширует все образы как OCI tarball’ы в `~/.coast/image-cache/`.
7. Если настроен `[coast.setup]`, собирает кастомный базовый образ DinD с указанными пакетами, командами и файлами.
8. Записывает директорию артефакта сборки с манифестом, разрешённым coastfile, переписанным compose и внедрёнными файлами.
9. Обновляет симлинк `latest`, чтобы он указывал на новую сборку.
10. Автоматически удаляет старые сборки сверх лимита хранения.

## Где хранятся сборки

```text
~/.coast/
  images/
    my-project/
      latest -> a3c7d783_20260227143000       (symlink)
      a3c7d783_20260227143000/                (versioned build)
        manifest.json
        coastfile.toml
        compose.yml
        inject/
      b4d8e894_20260226120000/                (older build)
        ...
  image-cache/                                (shared tarball cache)
    postgres_16_a1b2c3d4e5f6.tar
    redis_7_f6e5d4c3b2a1.tar
    coast-built_my-project_web_latest_...tar
```

Каждая сборка получает уникальный **ID сборки** в формате `{coastfile_hash}_{YYYYMMDDHHMMSS}`. Хэш включает содержимое Coastfile и разрешённую конфигурацию, поэтому изменения в Coastfile приводят к новому ID сборки.

Симлинк `latest` всегда указывает на самую свежую сборку для быстрого выбора. Если ваш проект использует типизированные Coastfile’ы (например, `Coastfile.light`), у каждого типа есть свой симлинк: `latest-light`.

Кэш образов в `~/.coast/image-cache/` общий для всех проектов. Если два проекта используют один и тот же образ Postgres, tarball кэшируется один раз.

## Что содержит сборка

Каждая директория сборки содержит:

- **`manifest.json`** -- полные метаданные сборки: имя проекта, временная метка сборки, хэш coastfile, список кэшированных/собранных образов, имена секретов, пропущенные сервисы, [стратегии томов](VOLUMES.md) и многое другое.
- **`coastfile.toml`** -- разрешённый Coastfile (объединённый с родительским, если используется `extends`).
- **`compose.yml`** -- переписанная версия вашего compose-файла, где директивы `build:` заменены на теги заранее собранных образов, а пропущенные сервисы удалены.
- **`inject/`** -- копии файлов хоста из `[inject].files` (например, `~/.gitconfig`, `~/.npmrc`).

## Сборки не содержат секретов

Секреты извлекаются на этапе сборки, но сохраняются в отдельном зашифрованном хранилище ключей по пути `~/.coast/keystore.db` -- не внутри директории артефакта сборки. Манифест записывает только **имена** извлечённых секретов, но никогда не значения.

Это означает, что артефакты сборки безопасно просматривать, не раскрывая конфиденциальные данные. Секреты расшифровываются и внедряются позже, когда создаётся экземпляр Coast с помощью `coast run`.

## Сборки и Docker

Сборка задействует три вида Docker-образов:

- **Собранные образы** -- compose-сервисы с директивами `build:` собираются на хосте через `docker build`, помечаются тегом `coast-built/{project}/{service}:latest` и сохраняются как tarball’ы в кэш образов.
- **Скачанные образы** -- compose-сервисы с директивами `image:` скачиваются и сохраняются как tarball’ы.
- **Образ Coast** -- если настроен `[coast.setup]`, поверх `docker:dind` собирается кастомный Docker-образ с указанными пакетами, командами и файлами. Помечается тегом `coast-image/{project}:{build_id}`.

Во время выполнения (`coast run`) эти tarball’ы загружаются во внутренний [демон DinD](RUNTIMES_AND_SERVICES.md) через `docker load`. Именно это позволяет экземплярам Coast быстро запускаться без необходимости скачивать образы из реестра.

## Сборки и экземпляры

Когда вы запускаете `coast run`, Coast выбирает последнюю сборку (или конкретную по `--build-id`) и использует её артефакты для создания экземпляра. ID сборки записывается в экземпляр.

Вам не нужно пересобирать, чтобы создавать больше экземпляров. Одна сборка может обслуживать множество экземпляров Coast, работающих параллельно.

## Когда нужно пересобирать

Пересобирайте только тогда, когда меняется ваш Coastfile, `docker-compose.yml` или конфигурация инфраструктуры. Пересборка ресурсозатратна -- она заново скачивает образы, заново собирает Docker-образы и заново извлекает секреты.

Изменения кода не требуют пересборки. Coast монтирует директорию проекта напрямую в каждый экземпляр, поэтому обновления кода подхватываются сразу.

## Автоматическая очистка

Coast хранит до 5 сборок на тип Coastfile. После каждого успешного `coast build` более старые сборки сверх лимита автоматически удаляются.

Сборки, которые используются запущенными экземплярами, никогда не удаляются, независимо от лимита. Если у вас 7 сборок, но 3 из них поддерживают активные экземпляры, все 3 защищены.

## Ручное удаление

Вы можете удалять сборки вручную с помощью `coast rm-build` или через вкладку Builds в Coastguard.

- **Полное удаление проекта** (`coast rm-build <project>`) требует, чтобы все экземпляры были сначала остановлены и удалены. Оно удаляет всю директорию сборок, связанные Docker-образы, тома и контейнеры.
- **Выборочное удаление** (по ID сборки, доступно в интерфейсе Coastguard) пропускает сборки, которые используются запущенными экземплярами.

## Типизированные сборки

Если ваш проект использует несколько Coastfile’ов (например, `Coastfile` для конфигурации по умолчанию и `Coastfile.snap` для томов, заполненных из снапшота), каждый тип поддерживает свой симлинк `latest-{type}` и свой собственный пул очистки из 5 сборок.

```bash
coast build              # uses Coastfile, updates "latest"
coast build --type snap  # uses Coastfile.snap, updates "latest-snap"
```

Очистка сборок типа `snap` никогда не затрагивает сборки `default`, и наоборот.
