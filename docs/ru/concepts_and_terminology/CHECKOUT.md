# Checkout

Checkout управляет тем, какой экземпляр Coast владеет [каноническими портами](PORTS.md) вашего проекта. Когда вы выполняете checkout Coast, `localhost:3000`, `localhost:5432` и любой другой канонический порт напрямую отображается на этот экземпляр.

```bash
coast checkout dev-1
```

```text
Before checkout:
  localhost:3000  ──→  (nothing)
  localhost:5432  ──→  (nothing)

After checkout:
  localhost:3000  ──→  dev-1 web
  localhost:5432  ──→  dev-1 db
```

Переключение checkout происходит мгновенно — Coast завершает и заново запускает лёгкие форвардеры `socat`. Никакие контейнеры не перезапускаются.

```bash
coast checkout dev-2   # instant swap

# localhost:3000  ──→  dev-2 web
# localhost:5432  ──→  dev-2 db
```

## Нужно ли выполнять Check Out?

Не обязательно. У каждого запущенного Coast всегда есть свои динамические порты, и вы можете обращаться к любому Coast через эти порты в любое время, не выполняя никакого checkout.

```bash
coast ports dev-1

# SERVICE    CANONICAL  DYNAMIC
# ★ web      3000       62217
#   db       5432       55681
```

Вы можете открыть `localhost:62217` в браузере, чтобы попасть на веб-сервер dev-1, не выполняя checkout. Для многих рабочих процессов это полностью нормально, и вы можете запускать столько Coast, сколько захотите, вообще никогда не используя `coast checkout`.

## Когда Checkout Полезен

Бывают ситуации, когда динамических портов недостаточно и вам нужны канонические порты:

- **Клиентские приложения, жёстко привязанные к каноническим портам.** Если у вас есть клиент, работающий вне Coast — фронтенд dev-сервер на хосте, мобильное приложение на телефоне или десктопное приложение — который ожидает `localhost:3000` или `localhost:8080`, менять номера портов повсюду непрактично. Checkout Coast даёт вам реальные порты без изменения какой-либо конфигурации.

- **Вебхуки и callback URL.** Сервисы вроде Stripe, GitHub или OAuth-провайдеров отправляют обратные вызовы на URL, который вы зарегистрировали — обычно что-то вроде `https://your-ngrok-tunnel.io`, который проксирует на `localhost:3000`. Если вы переключитесь на динамический порт, колбэки перестанут приходить. Checkout гарантирует, что канонический порт активен для Coast, который вы тестируете.

- **Инструменты для баз данных, отладчики и интеграции IDE.** Многие GUI-клиенты (pgAdmin, DataGrip, TablePlus), отладчики и конфигурации запуска в IDE сохраняют профили подключения с конкретным портом. Checkout позволяет сохранить ваши сохранённые профили и просто менять, какой Coast находится за ними — без перенастройки цели attach для отладчика или подключения к базе данных каждый раз при смене контекста.

## Освобождение Checkout

Если вы хотите освободить канонические порты, не выполняя checkout другого Coast:

```bash
coast checkout --none
```

После этого ни один Coast не владеет каноническими портами. Все Coasts остаются доступными через свои динамические порты.

## Только Один за Раз

Ровно один Coast может быть checked out в каждый момент времени. Если `dev-1` checked out и вы выполняете `coast checkout dev-2`, канонические порты мгновенно переключаются на `dev-2`. Разрыва нет — старые форвардеры завершаются и новые запускаются в рамках одной и той же операции.

```text
┌──────────────────────────────────────────────────┐
│  Your machine                                    │
│                                                  │
│  Canonical (checked-out Coast only):             │
│    localhost:3000 ──→ dev-2 web                  │
│    localhost:5432 ──→ dev-2 db                   │
│                                                  │
│  Dynamic (always available):                     │
│    localhost:62217 ──→ dev-1 web                 │
│    localhost:55681 ──→ dev-1 db                  │
│    localhost:63104 ──→ dev-2 web                 │
│    localhost:57220 ──→ dev-2 db                  │
└──────────────────────────────────────────────────┘
```

Динамические порты не затрагиваются checkout. Меняется только то, куда указывают канонические порты.
