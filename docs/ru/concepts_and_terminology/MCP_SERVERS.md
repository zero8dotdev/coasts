# MCP Servers

Серверы MCP (Model Context Protocol) дают ИИ-агентам доступ к инструментам — поиск по файлам, запросы к базам данных, поиск по документации, автоматизация браузера и многое другое. Coast может устанавливать и настраивать MCP-серверы внутри контейнера Coast, чтобы агент в контейнере имел доступ к нужным инструментам.

**Это актуально только если вы запускаете агента внутри контейнера Coast.** Если вы запускаете агентов на хосте (рекомендуемый подход), ваши MCP-серверы тоже запускаются на хосте и никакая из этих настроек не нужна. Эта страница опирается на [Agent Shells](AGENT_SHELLS.md) и добавляет поверх этого ещё один уровень сложности. Прежде чем продолжать, прочитайте предупреждения там.

## Internal vs Host-Proxied Servers

Coast поддерживает два режима для MCP-серверов, управляемые полем `proxy` в секции `[mcp]` вашего Coastfile.

### Internal Servers

Внутренние серверы устанавливаются и запускаются внутри контейнера DinD по пути `/mcp/<name>/`. У них есть прямой доступ к файловой системе контейнера и запущенным сервисам.

```toml
[mcp.context7]
install = "npm install -g @upstash/context7-mcp"
command = "npx"
args = ["-y", "@upstash/context7-mcp"]
```

Вы также можете копировать исходные файлы из вашего проекта в каталог MCP:

```toml
[mcp.my-custom-tool]
source = "tools/my-mcp-server"
install = ["npm install", "npm run build"]
command = "node"
args = ["dist/index.js"]
```

Поле `source` копирует файлы из `/workspace/<path>/` в `/mcp/<name>/` во время настройки. Команды `install` выполняются внутри этого каталога. Это полезно для MCP-серверов, которые живут в вашем репозитории.

### Host-Proxied Servers

Серверы, проксируемые с хоста, запускаются на вашей хост-машине, а не внутри контейнера. Coast генерирует конфиг клиента, который использует `coast-mcp-proxy` для проксирования MCP-запросов из контейнера на хост по сети.

```toml
[mcp.browser]
proxy = "host"
command = "npx"
args = ["@anthropic-ai/browser-mcp"]
```

У проксируемых с хоста серверов не может быть полей `install` или `source` — предполагается, что они уже доступны на хосте. Используйте этот режим для MCP-серверов, которым нужен доступ на уровне хоста, например для автоматизации браузера или инструментов работы с файловой системой хоста.

### When to Use Which

| Mode | Runs in | Good for | Limitations |
|---|---|---|---|
| Internal | контейнер DinD | Инструменты, которым нужен доступ к файловой системе контейнера, инструменты, специфичные для проекта | Должны устанавливаться в Alpine Linux, увеличивает время `coast run` |
| Host-proxied | хост-машина | Автоматизация браузера, инструменты уровня хоста, крупные заранее установленные серверы | Не могут напрямую обращаться к файловой системе контейнера |

## Client Connectors

Секция `[mcp_clients]` сообщает Coast, куда записывать сгенерированную конфигурацию MCP-серверов, чтобы агент внутри контейнера мог обнаружить серверы.

### Built-in Formats

Для Claude Code и Cursor достаточно пустой секции с правильным именем — Coast автоматически определяет формат и путь конфигурации по умолчанию:

```toml
[mcp_clients.claude-code]
# Writes to /root/.claude/mcp_servers.json (auto-detected)

[mcp_clients.cursor]
# Writes to /workspace/.cursor/mcp.json (auto-detected)
```

### Custom Config Path

Для других AI-инструментов явно укажите формат и путь:

```toml
[mcp_clients.my-tool]
format = "claude-code"
config_path = "/home/coast/.config/my-tool/mcp.json"
```

### Command-Based Connectors

Вместо записи файла вы можете передать сгенерированный JSON конфигурации в команду через stdin:

```toml
[mcp_clients.custom-setup]
run = "my-config-tool import-mcp --stdin"
```

Поле `run` взаимоисключающе с `format` и `config_path`.

## Coastguard MCP Tab

Веб-интерфейс [Coastguard](COASTGUARD.md) предоставляет видимость вашей MCP-конфигурации на вкладке MCP.

![MCP tab in Coastguard](../../assets/coastguard-mcp.png)
*Вкладка MCP в Coastguard, показывающая настроенные серверы, их инструменты и расположения клиентских конфигураций.*

Вкладка содержит три секции:

- **MCP Servers** — перечисляет каждый объявленный сервер с его именем, типом (Internal или Host), командой и статусом (Installed, Proxied или Not Installed).
- **Tools** — выберите сервер, чтобы просмотреть инструменты, которые он предоставляет через протокол MCP. Для каждого инструмента показаны имя и описание; перейдите внутрь, чтобы увидеть полную схему входных данных.
- **Client Locations** — показывает, куда были записаны сгенерированные конфигурационные файлы (например, формат `claude-code` в `/root/.claude/mcp_servers.json`).

## CLI Commands

```bash
coast mcp dev-1 ls                          # list servers with type and status
coast mcp dev-1 tools context7              # list tools exposed by a server
coast mcp dev-1 tools context7 info resolve # show input schema for a specific tool
coast mcp dev-1 locations                   # show where client configs were written
```

Команда `tools` работает, отправляя JSON-RPC запросы `initialize` и `tools/list` процессу MCP-сервера внутри контейнера. Она работает только для внутренних серверов — серверы, проксируемые с хоста, нужно инспектировать с хоста.

## How Installation Works

Во время `coast run`, после того как внутренний Docker daemon готов и сервисы запускаются, Coast настраивает MCP:

1. Для каждого **внутреннего** MCP-сервера:
   - Создаёт `/mcp/<name>/` внутри контейнера DinD
   - Если задан `source`, копирует файлы из `/workspace/<source>/` в `/mcp/<name>/`
   - Выполняет каждую команду `install` внутри `/mcp/<name>/` (например, `npm install -g @upstash/context7-mcp`)

2. Для каждого **клиентского коннектора**:
   - Генерирует JSON-конфиг в соответствующем формате (Claude Code или Cursor)
   - Для внутренних серверов подставляет их реальные `command` и `args`, устанавливая `cwd` в `/mcp/<name>/`
   - Для серверов, проксируемых с хоста, задаёт `coast-mcp-proxy` как команду, а имя сервера — как аргумент
   - Записывает конфиг по целевому пути (или передаёт его в команду `run`)

Проксируемые с хоста серверы полагаются на `coast-mcp-proxy` внутри контейнера, чтобы пересылать запросы протокола MCP обратно на хост-машину, где запускается реальный процесс MCP-сервера.

## Full Example

Coastfile, который настраивает внутренний инструмент для документации и проксируемый с хоста браузерный инструмент, подключённые к Claude Code:

```toml
[mcp.context7]
install = "npm install -g @upstash/context7-mcp"
command = "npx"
args = ["-y", "@upstash/context7-mcp"]

[mcp.browser]
proxy = "host"
command = "npx"
args = ["@anthropic-ai/browser-mcp"]

[mcp_clients.claude-code]
```

После `coast run` Claude Code внутри контейнера видит оба сервера в своей MCP-конфигурации — `context7`, запущенный локально в `/mcp/context7/`, и `browser`, проксируемый на хост.

## Agents Running on the Host

Если ваш кодинговый агент работает на хост-машине (рекомендуемый подход), ваши MCP-серверы тоже работают на хосте, и конфигурация Coast `[mcp]` не используется. Однако есть один момент: **MCP-серверы, которые подключаются к базам данных или сервисам внутри Coast, должны знать правильный порт.**

Когда сервисы работают внутри Coast, они доступны на динамических портах, которые меняются каждый раз при запуске нового экземпляра. MCP для базы данных на хосте, который подключается к `localhost:5432`, сможет достучаться только до базы данных Coast, которая [checked-out](CHECKOUT.md) — или ни до чего, если никакой Coast не checked-out. Для экземпляров без checked-out вам нужно было бы перенастроить MCP на использование [dynamic port](PORTS.md) (например, `localhost:55681`).

Есть два способа обойти это:

**Используйте общие сервисы.** Если ваша база данных работает как [shared service](SHARED_SERVICES.md), она живёт в Docker daemon хоста на своём каноническом порту (`localhost:5432`). Каждый экземпляр Coast подключается к ней через bridge-сеть, а ваш MCP на хосте подключается к той же базе данных на том же порту, как и всегда. Никакой перенастройки, никакого поиска динамического порта. Это самый простой подход.

**Используйте `coast exec` или `coast docker`.** Если ваша база данных работает внутри Coast (изолированные тома), ваш агент на хосте всё равно может делать к ней запросы, выполняя команды через Coast (см. [Exec & Docker](EXEC_AND_DOCKER.md)):

```bash
coast exec dev-1 -- psql -h localhost -U myuser -d mydb -c "SELECT count(*) FROM users"
coast docker dev-1 exec -i my-postgres psql -U myuser -d mydb -c "\\dt"
```

Это позволяет вообще не знать динамический порт — команда выполняется внутри Coast, где база данных находится на своём каноническом порту.

Для большинства рабочих процессов общие сервисы — путь наименьшего сопротивления. Ваша конфигурация MCP на хосте остаётся точно такой же, какой была до того, как вы начали использовать Coasts.
