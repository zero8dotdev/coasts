# Оптимизация производительности

Coast спроектирован так, чтобы переключение веток было быстрым, но в больших монорепозиториях поведение по умолчанию всё равно может добавлять задержки. На этой странице описаны рычаги, доступные в вашем Coastfile, и, что важнее, на какие части `coast assign` они реально влияют.

## Почему Assign может быть медленным

`coast assign` делает несколько вещей при переключении Coast на новый worktree:

```text
coast assign dev-1 --worktree feature/payments

  1. classify services and optional rebuild-trigger diff
  2. stop affected services
  3. create git worktree (if new)
  4. bootstrap gitignored files into the worktree (first assign only)
  5. remount /workspace
  6. recreate/restart containers
  7. rebuild images for services using "rebuild"
  8. wait for healthy
```

Наиболее переменные по стоимости операции обычно — это **первичное bootstrap-копирование gitignored файлов**, **перезапуски контейнеров** и **пересборки образов**. Опциональный diff веток, используемый для триггеров пересборки, значительно дешевле, но он тоже может накапливаться, если вы укажете слишком широкие наборы триггеров.

### Bootstrap gitignored файлов

Когда worktree создаётся впервые, Coast выполняет bootstrap выбранных gitignored файлов из корня проекта в этот worktree.

Последовательность следующая:

1. Запустить `git ls-files --others --ignored --exclude-standard` на хосте, чтобы перечислить игнорируемые файлы.
2. Отфильтровать распространённые тяжёлые директории плюс любые настроенные `exclude_paths`.
3. Запустить `rsync --files-from` с `--link-dest`, чтобы выбранные файлы были захардлинкованы в worktree вместо побайтового копирования.
4. Записать успешный bootstrap во внутренние метаданные worktree, чтобы последующие assign в тот же worktree могли его пропустить.

Если `rsync` недоступен, Coast откатывается на пайплайн `tar`.

Крупные директории вроде `node_modules`, `.git`, `dist`, `target`, `.next`, `.nuxt`, `.cache`, `.worktrees` и `.coasts` исключаются автоматически. Ожидается, что большие директории зависимостей будут обрабатываться кэшами сервисов или томами (volumes), а не этим универсальным шагом bootstrap.

Поскольку список файлов генерируется заранее, `rsync` работает по целевому списку, а не вслепую обходит весь репозиторий. Тем не менее репозитории с очень большими наборами игнорируемых файлов всё равно могут платить заметную разовую стоимость bootstrap при первом создании worktree. Если вам когда-нибудь понадобится обновить этот bootstrap вручную, выполните `coast assign --force-sync`.

### Diff для триггеров пересборки

Coast вычисляет diff веток только когда настроен `[assign.rebuild_triggers]`. В этом случае он выполняет:

```bash
git diff --name-only <previous>..<worktree>
```

Результат используется для понижения стратегии сервиса с `rebuild` до `restart`, когда ни один из его trigger-файлов не изменился.

Это намного уже, чем старая модель «diff каждого отслеживаемого файла при каждом assign». Если вы не настраиваете триггеры пересборки, шага diff веток здесь вообще нет.

`exclude_paths` сейчас не изменяет этот diff. Держите списки триггеров сфокусированными на реальных входных данных времени сборки, таких как Dockerfile’ы, lockfile’ы и манифесты пакетов.

## `exclude_paths` — главный рычаг для новых worktree

Опция `exclude_paths` в вашем Coastfile говорит Coast пропускать целые деревья директорий при построении списка файлов для bootstrap gitignored файлов в новом worktree.

```toml
[assign]
default = "none"
exclude_paths = [
    "docs",
    "scripts",
    "test-fixtures",
    "apps/mobile",
]
```

Файлы под исключёнными путями всё равно присутствуют в worktree, если Git их отслеживает. Coast просто избегает затрат времени на перечисление и хардлинкование игнорируемых файлов под этими деревьями во время первичного bootstrap.

Это наиболее заметно, когда в корне репозитория есть большие игнорируемые директории, которые вашим работающим сервисам не нужны: несвязанные приложения, вендоренные кэши, тестовые фикстуры, сгенерированная документация и другие тяжёлые деревья.

Если вы многократно выполняете assign в один и тот же уже синхронизированный worktree, `exclude_paths` менее важен, потому что bootstrap пропускается. В этом случае доминирующим фактором становится выбор между перезапуском и пересборкой сервисов.

### Как выбирать, что исключать

Начните с профилирования игнорируемых файлов:

```bash
git ls-files --others --ignored --exclude-standard | cut -d'/' -f1 | sort | uniq -c | sort -rn
```

Если вам также нужен вид по отслеживаемой структуре для настройки триггеров пересборки, используйте:

```bash
git ls-files | cut -d'/' -f1 | sort | uniq -c | sort -rn
```

**Оставляйте** директории, которые:
- Содержат исходный код, примонтированный в работающие сервисы
- Содержат общие библиотеки, импортируемые этими сервисами
- Содержат сгенерированные файлы или кэши, которые вашему рантайму реально нужны при первом запуске
- Упоминаются в `[assign.rebuild_triggers]`

**Исключайте** директории, которые:
- Относятся к приложениям или сервисам, не запускающимся в вашем Coast
- Содержат документацию, скрипты, конфиги CI или инструменты, не относящиеся к рантайму
- Держат большие игнорируемые кэши, которые уже сохраняются где-то ещё, например в выделенных кэшах сервисов или общих томах

### Пример: монорепозиторий с несколькими приложениями

Монорепозиторий со многими директориями верхнего уровня, но только часть из них важна для сервисов, работающих в этом Coast:

```text
  13,000  bookface/         ← active
   7,000  ycinternal/       ← active
     850  shared/           ← used by both
   3,800  .yarn/            ← excludable
   2,500  startupschool/    ← excludable
     500  misc/             ← excludable
     300  ycapp/            ← excludable
     ...  (12 more dirs)    ← excludable
```

```toml
[assign]
default = "none"
exclude_paths = [
    ".yarn",
    "startupschool",
    "misc",
    "ycapp",
    "apply",
    "cli",
    "deploy",
    "lambdas",
    # ... any other directories not needed by active services
]
```

Это оставляет первичный bootstrap worktree сфокусированным на директориях, которые реально нужны работающим сервисам, вместо того чтобы тратить время на несвязанные игнорируемые деревья.

## Уберите неактивные сервисы из `[assign.services]`

Если ваш `COMPOSE_PROFILES` запускает только подмножество сервисов, удалите неактивные сервисы из `[assign.services]`. Coast вычисляет стратегию assign для каждого перечисленного сервиса, и перезапуск или пересборка сервиса, который не запущен, — это лишняя работа.

```toml
# Bad — restarts services that aren't running
[assign.services]
web = "restart"
api = "restart"
mobile-api = "restart"   # not in COMPOSE_PROFILES
batch-worker = "restart"  # not in COMPOSE_PROFILES

# Good — only services that are actually running
[assign.services]
web = "restart"
api = "restart"
```

То же относится и к `[assign.rebuild_triggers]` — удалите записи для сервисов, которые не активны.

## Используйте `"hot"` где возможно

Стратегия `"hot"` полностью пропускает перезапуск контейнера. [Перемонтирование файловой системы](FILESYSTEM.md) подменяет код под `/workspace`, а файловый watcher сервиса (Vite, webpack, nodemon, air и т. п.) автоматически подхватывает изменения.

```toml
[assign.services]
web = "hot"        # Vite/webpack dev server with HMR
api = "restart"    # Rails/Go — needs a process restart
```

`"hot"` быстрее, чем `"restart"`, потому что избегает цикла stop/start контейнера. Используйте её для любого сервиса, который запускает dev-сервер с отслеживанием файлов. Оставляйте `"restart"` для сервисов, которые загружают код при старте и не следят за изменениями (большинство приложений на Rails, Go и Java).

## Используйте `"rebuild"` с триггерами

Если стратегия сервиса по умолчанию — `"rebuild"`, каждое переключение ветки пересобирает Docker-образ — даже если ничего, влияющего на образ, не менялось. Добавьте `[assign.rebuild_triggers]`, чтобы ограничить пересборку конкретными файлами:

```toml
[assign.services]
worker = "rebuild"

[assign.rebuild_triggers]
worker = ["Dockerfile", "package.json", "package-lock.json"]
```

Если ни один из trigger-файлов не изменился между ветками, Coast пропускает пересборку и вместо этого откатывается к перезапуску. Это позволяет избежать дорогих сборок образов при обычных изменениях кода.

## Итоги

| Оптимизация | Влияние | На что влияет | Когда использовать |
|---|---|---|---|
| `exclude_paths` | Высокое | первичный bootstrap gitignored файлов | Репозитории с большими игнорируемыми деревьями, которые вашему Coast не нужны |
| Удаление неактивных сервисов | Среднее | перезапуск/пересоздание сервисов | Когда `COMPOSE_PROFILES` ограничивает, какие сервисы запускаются |
| Стратегия `"hot"` | Высокое | перезапуск контейнера | Сервисы с файловыми watcher’ами (Vite, webpack, nodemon, air) |
| `rebuild_triggers` | Высокое | пересборки образов + опциональный diff веток | Сервисы, использующие `"rebuild"`, которым это нужно только при инфраструктурных изменениях |

Если новые worktree медленно назначаются при первом assign, начните с `exclude_paths`. Если повторные assign медленные, сфокусируйтесь на `hot` vs `restart`, уберите неактивные сервисы и держите `rebuild_triggers` узкими.
