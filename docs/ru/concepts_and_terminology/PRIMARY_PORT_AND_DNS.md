# Основной порт и DNS

Основной порт — это необязательная функция для удобства, которая создаёт быструю ссылку на один из ваших сервисов — обычно на ваш веб‑фронтенд. Он отображается как кликабельный бейдж в Coastguard и как отмеченная звёздочкой запись в `coast ports`. Он не меняет то, как работают порты; он просто выбирает один, который нужно выделить.

## Настройка основного порта

Добавьте `primary_port` в секцию `[coast]` вашего Coastfile, сославшись на ключ из [`[ports]`](PORTS.md):

```toml
[coast]
name = "my-app"
primary_port = "web"

[ports]
web = 3000
backend = 8080
```

Если в вашем проекте только один порт, Coast автоматически определяет его как основной — вам не нужно задавать его явно.

Вы также можете переключать основной порт во вкладке Ports в Coastguard, нажимая на иконку звезды рядом с любым сервисом, или из CLI с помощью `coast ports set-primary`. Настройка задаётся на уровне сборки, поэтому все экземпляры, созданные из одной и той же сборки, разделяют один и тот же основной порт.

## Что это даёт

```text
coast ports dev-1

  SERVICE    CANONICAL  DYNAMIC
  ★ web      3000       62217
    backend  8080       63889
```

Сервис со звёздочкой — ваш основной. В Coastguard он появляется как кликабельный бейдж рядом с именем экземпляра — один клик открывает ваше приложение в браузере.

Это особенно полезно для:

- **Агентов на стороне хоста** — дайте вашему AI‑агенту один URL, по которому можно проверять изменения. Вместо того чтобы говорить ему «открой localhost:62217», URL основного порта доступен программно из `coast ls` и API демона.
- **Browser MCP** — если ваш агент использует browser MCP для проверки UI‑изменений, URL основного порта является канонической целью, на которую следует его направлять.
- **Быстрой итерации** — доступ в один клик к сервису, на который вы смотрите чаще всего.

Основной порт полностью необязателен. Всё работает и без него — это функция качества жизни для более быстрой навигации.

## Маршрутизация по поддоменам

Когда вы запускаете несколько экземпляров Coast с изолированными базами данных, в браузере они все используют `localhost`. Это означает, что cookie, установленные `localhost:62217` (dev-1), видны для `localhost:63104` (dev-2). Если ваше приложение использует cookie сессии, вход в один экземпляр может мешать другому.

Маршрутизация по поддоменам решает это, давая каждому экземпляру собственный origin:

```text
Без маршрутизации по поддоменам:
  dev-1 web  →  http://localhost:62217
  dev-2 web  →  http://localhost:63104
  (cookie общие — оба это "localhost")

С маршрутизацией по поддоменам:
  dev-1 web  →  http://dev-1.localhost:62217
  dev-2 web  →  http://dev-2.localhost:63104
  (cookie изолированы — разные поддомены)
```

Включите это для проекта во вкладке Ports в Coastguard (переключатель внизу страницы) или через API настроек демона.

### Компромисс: CORS

Недостаток в том, что вашему приложению могут понадобиться корректировки CORS. Если ваш фронтенд на `dev-1.localhost:3000` делает API‑запросы на `dev-1.localhost:8080`, браузер считает это запросами между источниками (cross-origin), потому что порт отличается. Большинство dev‑серверов уже с этим справляются, но если после включения маршрутизации по поддоменам вы видите ошибки CORS, проверьте конфигурацию разрешённых источников (allowed origins) вашего приложения.

## Шаблоны URL

У каждого сервиса есть шаблон URL, который управляет тем, как генерируются его ссылки. По умолчанию это:

```text
http://localhost:<port>
```

Плейсхолдер `<port>` заменяется на фактический номер порта — канонический порт, когда экземпляр [выдан](CHECKOUT.md), или динамический порт в противном случае. Когда включена маршрутизация по поддоменам, `localhost:` заменяется на `{instance}.localhost:`.

Вы можете настраивать шаблоны для каждого сервиса во вкладке Ports в Coastguard (иконка карандаша рядом с каждым сервисом). Это полезно, если ваш dev‑сервер использует HTTPS, пользовательское имя хоста или нестандартную схему URL:

```text
https://my-service.localhost:<port>
```

Шаблоны хранятся в настройках демона и сохраняются между перезапусками.

## Настройка DNS

Большинство браузеров из коробки резолвят `*.localhost` в `127.0.0.1`, поэтому маршрутизация по поддоменам работает без какой‑либо настройки DNS.

Если вам нужно разрешение пользовательского домена (например, `*.localcoast`), Coast включает встроенный DNS‑сервер. Настройте его один раз:

```bash
coast dns setup    # writes /etc/resolver/localcoast (requires sudo)
coast dns status   # check if DNS is configured
coast dns remove   # remove the resolver entry
```

Это необязательно и требуется только если `*.localhost` не работает в вашем браузере или вы хотите пользовательский TLD.
