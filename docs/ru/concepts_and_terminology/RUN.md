# Запуск

`coast run` создаёт новый экземпляр Coast. Он определяет последнюю [сборку](BUILDS.md), подготавливает [DinD-контейнер](RUNTIMES_AND_SERVICES.md), загружает кэшированные образы, запускает ваши сервисы compose, выделяет [динамические порты](PORTS.md) и записывает экземпляр в базу данных состояния.

```bash
coast run dev-1
```

Если передать `-w`, Coast также [назначает](ASSIGN.md) рабочее дерево после завершения подготовки:

```bash
coast run dev-1 -w feature/oauth
```

Это наиболее распространённый сценарий, когда harness или агент создаёт рабочее дерево и за один шаг нуждается в Coast для него.

## Что происходит

`coast run` выполняет четыре фазы:

1. **Проверка и вставка** — проверяет, что имя уникально, определяет ID сборки (из симлинка `latest` или из явно указанного `--build-id`) и вставляет запись экземпляра `Provisioning`.
2. **Подготовка Docker** — создаёт DinD-контейнер в хостовом демоне, собирает все образы для конкретного экземпляра, загружает кэшированные tarball-файлы образов во внутренний демон, переписывает compose-файл, внедряет секреты и выполняет `docker compose up -d`.
3. **Завершение** — сохраняет выделения портов, устанавливает основной порт, если он ровно один, и переводит экземпляр в состояние `Running`.
4. **Необязательное назначение рабочего дерева** — если был указан `-w <worktree>`, выполняет `coast assign` для нового экземпляра. Если назначение завершается ошибкой, Coast всё равно продолжает работать — ошибка записывается как предупреждение.

Постоянный том `/var/lib/docker` внутри DinD-контейнера означает, что при последующих запусках загрузка образов пропускается. Новый `coast run` с холодными кэшами может занимать 20+ секунд; повторный запуск после `coast rm` обычно завершается менее чем за 10 секунд.

## Использование CLI

```text
coast run <name> [options]
```

| Флаг | Описание |
|------|-------------|
| `-w`, `--worktree <name>` | Назначить это рабочее дерево после завершения подготовки |
| `--n <count>` | Пакетное создание. Имя должно содержать `{n}` (например, `coast run dev-{n} --n=5` создаёт dev-1 по dev-5) |
| `-t`, `--type <type>` | Использовать типизированную сборку (например, `--type snap` определяет `latest-snap` вместо `latest`) |
| `--force-remove-dangling` | Удалить оставшийся Docker-контейнер с тем же именем перед созданием |
| `-s`, `--silent` | Подавить вывод прогресса; печатать только итоговую сводку или ошибки |
| `-v`, `--verbose` | Показывать подробные детали, включая логи сборки Docker |

Git-ветка всегда автоматически определяется из текущего HEAD.

## Пакетное создание

Используйте `{n}` в имени и `--n`, чтобы создать сразу несколько экземпляров:

```bash
coast run dev-{n} --n=5
```

Это последовательно создаёт `dev-1`, `dev-2`, `dev-3`, `dev-4`, `dev-5`. Каждый экземпляр получает собственный DinD-контейнер, выделения портов и состояние томов. Для пакетов размером более 10 требуется подтверждение.

## Типизированные сборки

Если ваш проект использует несколько типов Coastfile (см. [Типы Coastfile](COASTFILE_TYPES.md)), передайте `--type`, чтобы выбрать, какую сборку использовать:

```bash
coast run dev-1                    # resolves "latest"
coast run test-1 --type test       # resolves "latest-test"
coast run snapshot-1 --type snap   # resolves "latest-snap"
```

## Run vs assign and remove

- `coast run` создаёт **новый** экземпляр. Используйте его, когда вам нужен ещё один Coast.
- `coast assign` переназначает **существующий** экземпляр на другое рабочее дерево. Используйте
  его, когда у вас уже есть Coast и вы хотите переключить, какой код он запускает.
- `coast rm` полностью удаляет экземпляр. Используйте его, когда хотите остановить
  Coasts или пересоздать один с нуля.

Для большинства повседневных переключений `coast rm` не нужен; обычно достаточно
`coast assign` и `coast checkout`. Обращайтесь к `coast rm`, когда хотите чистое
пересоздание, особенно после пересборки вашего Coastfile или сборки.

Их можно комбинировать: `coast run dev-3 -w feature/billing` создаёт экземпляр
и назначает рабочее дерево за один шаг.

## Висячие контейнеры

Если предыдущий `coast run` был прерван или `coast rm` не выполнил полную очистку, вы можете увидеть ошибку "dangling Docker container". Передайте `--force-remove-dangling`, чтобы удалить оставшийся контейнер и продолжить:

```bash
coast run dev-1 --force-remove-dangling
```
