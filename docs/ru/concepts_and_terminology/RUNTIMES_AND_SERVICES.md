# Рантаймы и сервисы

Coast работает внутри контейнерного рантайма — внешнего контейнера, который хостит собственный демон Docker (или Podman). Сервисы вашего проекта запускаются внутри этого внутреннего демона, полностью изолированные от других экземпляров Coast. В настоящее время **DinD (Docker-in-Docker) — единственный рантайм, протестированный для продакшена.** Сейчас мы рекомендуем придерживаться DinD, пока поддержка Podman и Sysbox не будет тщательно протестирована.

## Рантаймы

Поле `runtime` в вашем Coastfile выбирает, какой контейнерный рантайм используется для Coast. Значение по умолчанию — `dind`, и вы можете вовсе его опустить:

```toml
[coast]
name = "my-app"
runtime = "dind"
```

Допускаются три значения: `dind`, `sysbox` и `podman`. На практике только DinD подключён к демону и протестирован сквозным образом.

### DinD (Docker-in-Docker)

Рантайм по умолчанию и единственный, который вам стоит использовать сегодня. Coast создаёт контейнер из образа `docker:dind` с включённым режимом `--privileged`. Внутри этого контейнера запускается полноценный демон Docker, а ваши сервисы из `docker-compose.yml` запускаются как вложенные контейнеры.

DinD полностью интегрирован:

- Образы предварительно кэшируются на хосте и загружаются во внутренний демон при `coast run`
- Образы на экземпляр собираются на хосте и прокидываются через `docker save | docker load`
- Состояние внутреннего демона сохраняется в именованном томе (`coast-dind--{project}--{instance}`) по пути `/var/lib/docker`, поэтому последующие запуски полностью пропускают загрузку образов
- Порты публикуются напрямую из контейнера DinD на хост
- Переопределения Compose, мостинг общей сети сервисов, инъекция секретов и стратегии томов — всё работает

### Sysbox (в будущем)

Sysbox — это OCI-рантайм только для Linux, который предоставляет rootless-контейнеры без `--privileged`. Он бы использовал `--runtime=sysbox-runc` вместо привилегированного режима, что обеспечивает лучшую модель безопасности. Реализация трейта существует в кодовой базе, но не подключена к демону. На macOS не работает.

### Podman (в будущем)

Podman заменил бы внутренний демон Docker на демон Podman, работающий внутри `quay.io/podman/stable`, используя `podman-compose` вместо `docker compose`. Реализация трейта существует, но не подключена к демону.

Когда поддержка Sysbox и Podman стабилизируется, эта страница будет обновлена. Пока оставьте `runtime` как `dind` или опустите его.

## Архитектура Docker-in-Docker

Каждый Coast — это вложенный контейнер. Хостовый демон Docker управляет внешним контейнером DinD, а внутренний демон Docker внутри него управляет вашими compose-сервисами.

```text
Host machine
│
├── Docker daemon (host)
│   │
│   ├── coast container: dev-1 (docker:dind, --privileged)
│   │   │
│   │   ├── Inner Docker daemon
│   │   │   ├── web        (your app, :3000)
│   │   │   ├── postgres   (database, :5432)
│   │   │   └── redis      (cache, :6379)
│   │   │
│   │   ├── /workspace          ← bind mount of your project root
│   │   ├── /image-cache        ← read-only mount of ~/.coast/image-cache/
│   │   ├── /coast-artifact     ← read-only mount of the build artifact
│   │   ├── /coast-override     ← generated compose overrides
│   │   └── /var/lib/docker     ← named volume (inner daemon state)
│   │
│   ├── coast container: dev-2 (docker:dind, --privileged)
│   │   └── (same structure, fully isolated)
│   │
│   └── shared postgres (host-level, bridge network)
│
└── ~/.coast/
    ├── image-cache/    ← OCI tarballs shared across all projects
    └── state.db        ← instance metadata
```

Когда `coast run` создаёт экземпляр, он:

1. Создаёт и запускает контейнер DinD в хостовом демоне
2. Опросом выполняет `docker info` внутри контейнера, пока внутренний демон не будет готов (до 120 секунд)
3. Проверяет, какие образы уже есть во внутреннем демоне (из постоянного тома `/var/lib/docker`) и загружает любые недостающие tarball’ы из кэша
4. Прокидывает образы на экземпляр, собранные на хосте, через `docker save | docker load`
5. Привязывает `/host-project` к `/workspace`, чтобы compose-сервисы видели ваш исходный код
6. Запускает `docker compose up -d` внутри контейнера и ждёт, пока все сервисы не будут запущены или healthy

Постоянный том `/var/lib/docker` — ключевая оптимизация. При свежем `coast run` загрузка образов во внутренний демон может занимать 20+ секунд. При последующих запусках (даже после `coast rm` и повторного запуска) во внутреннем демоне образы уже закэшированы, и старт сокращается до менее чем 10 секунд.

## Сервисы

Сервисы — это контейнеры (или процессы, в случае с [bare services](BARE_SERVICES.md)), работающие внутри вашего Coast. Для Coast на основе compose это сервисы, определённые в вашем `docker-compose.yml`.

![Services tab in Coastguard](../../assets/coastguard-services.png)
*Вкладка Services в Coastguard, показывающая compose-сервисы, их статус, образы и сопоставления портов.*

Вкладка Services в Coastguard показывает каждый сервис, запущенный внутри экземпляра Coast:

- **Service** — имя compose-сервиса (например, `web`, `backend`, `redis`). Перейдите внутрь, чтобы увидеть подробные данные inspect, логи и статистику для этого контейнера.
- **Status** — запущен ли сервис, остановлен или находится в состоянии ошибки.
- **Image** — Docker-образ, из которого собран сервис.
- **Ports** — исходные сопоставления портов из compose и управляемые coast [канонические/динамические порты](PORTS.md). Динамические порты всегда доступны; канонические порты маршрутизируются только на [checked-out](CHECKOUT.md) экземпляр.

Вы можете выбрать несколько сервисов и пакетно остановить, запустить, перезапустить или удалить их с панели инструментов.

Сервисы, настроенные как [shared services](SHARED_SERVICES.md), запускаются в хостовом демоне, а не внутри Coast, поэтому в этом списке они не отображаются. Для них есть отдельная вкладка.

## `coast ps`

CLI-эквивалент вкладки Services — это `coast ps`:

```bash
coast ps dev-1
```

```text
Services in coast instance 'dev-1':
  NAME                      STATUS               PORTS
  backend                   running              0.0.0.0:8080->8080/tcp, 0.0.0.0:40000->40000/tcp
  mailhog                   running              0.0.0.0:1025->1025/tcp, 0.0.0.0:8025->8025/tcp
  reach-web                 running              0.0.0.0:4000->4000/tcp
  test-redis                running              0.0.0.0:6380->6379/tcp
  web                       running              0.0.0.0:3000->3000/tcp
```

Под капотом демон выполняет `docker compose ps --format json` внутри контейнера DinD и парсит JSON-вывод. Результаты проходят через несколько фильтров перед тем, как вернуться:

- **Shared services** удаляются — они работают на хосте, а не внутри Coast.
- **Одноразовые джобы** (сервисы без портов) скрываются после успешного завершения. Если они падают, они отображаются, чтобы вы могли разобраться.
- **Отсутствующие сервисы** — если долгоживущий сервис, который должен присутствовать, отсутствует в выводе, он добавляется со статусом `down`, чтобы вы понимали, что что-то не так.

Для более глубокого инспектирования используйте `coast logs`, чтобы tail’ить вывод сервиса, и [`coast exec`](EXEC_AND_DOCKER.md), чтобы получить shell внутри контейнера Coast. См. [Logs](LOGS.md) для полного описания стриминга логов и компромисса MCP.

```bash
coast logs dev-1 --service web --tail 100
coast exec dev-1
```
