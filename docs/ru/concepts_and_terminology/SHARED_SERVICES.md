# Общие сервисы

Общие сервисы — это контейнеры баз данных и инфраструктуры (Postgres, Redis, MongoDB и т. д.), которые запускаются на вашем хостовом Docker-демоне, а не внутри Coast. Экземпляры Coast подключаются к ним через bridge-сеть, поэтому каждый Coast обращается к одному и тому же сервису на одном и том же томе хоста.

![Shared services in Coastguard](../../assets/coastguard-shared-services.png)
*Вкладка общих сервисов в Coastguard, показывающая управляемые на хосте Postgres, Redis и MongoDB.*

## Как это работает

Когда вы объявляете общий сервис в вашем Coastfile, Coast запускает его на хостовом демоне и удаляет его из compose-стека, который выполняется внутри каждого контейнера Coast. Затем Coasts настраиваются так, чтобы маршрутизировать свои подключения обратно на хост.

```text
Host Docker daemon
  |
  +--> postgres (host volume: infra_postgres_data)
  +--> redis    (host volume: infra_redis_data)
  +--> mongodb  (host volume: infra_mongodb_data)
  |
  +--> Coast: dev-1  --bridge network--> host postgres, redis, mongodb
  +--> Coast: dev-2  --bridge network--> host postgres, redis, mongodb
```

Поскольку общие сервисы повторно используют ваши существующие хостовые тома, любые данные, которые у вас уже есть после локального запуска `docker-compose up`, сразу доступны вашим Coast.

## Когда использовать общие сервисы

- В вашем проекте есть интеграции MCP, которые подключаются к локальной базе данных — общие сервисы позволяют им продолжать работать без перенастройки. MCP базы данных на вашем хосте, подключающийся к `localhost:5432`, продолжит работать, потому что общий Postgres находится на хосте на том же порту. Никакого динамического обнаружения портов, никакой перенастройки MCP. Подробнее см. [MCP Servers](MCP_SERVERS.md).
- Вам нужны более «лёгкие» экземпляры Coast, поскольку им не требуется запускать собственные контейнеры баз данных.
- Вам не нужна изоляция данных между экземплярами Coast (каждый экземпляр видит одни и те же данные).
- Вы запускаете coding agents на хосте (см. [Filesystem](FILESYSTEM.md)) и хотите, чтобы они имели доступ к состоянию базы данных без маршрутизации через [`coast exec`](EXEC_AND_DOCKER.md). С общими сервисами существующие инструменты работы с базой данных и MCP у агента работают без изменений.

См. страницу [Volume Topology](VOLUMES.md) для альтернатив, когда изоляция всё же нужна.

## Предупреждение о неоднозначности томов

Имена docker-томов не всегда глобально уникальны. Если вы запускаете `docker-compose up` из нескольких разных проектов, хостовые тома, которые Coast подключает к общим сервисам, могут оказаться не теми, которые вы ожидаете.

Перед запуском Coast с общими сервисами убедитесь, что последний выполненный вами `docker-compose up` был запущен из проекта, который вы собираетесь использовать с Coast. Это гарантирует, что хостовые тома соответствуют тому, что ожидает ваш Coastfile.

## Устранение неполадок

Если кажется, что ваши общие сервисы указывают на неправильный хостовый том:

1. Откройте UI [Coastguard](COASTGUARD.md) (`coast ui`).
2. Перейдите на вкладку **Shared Services**.
3. Выберите затронутые сервисы и нажмите **Remove**.
4. Нажмите **Refresh Shared Services**, чтобы пересоздать их из вашей текущей конфигурации Coastfile.

Это остановит и пересоздаст контейнеры общих сервисов, повторно подключив их к правильным хостовым томам.
