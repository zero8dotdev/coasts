Вы устанавливаете Coasts в этот проект. Coast (контейнеризованный хост) — это CLI-инструмент, который запускает несколько изолированных сред разработки на одной машине, используя контейнеры Docker-in-Docker. Каждая среда получает собственные порты, тома и среду выполнения — идеально для параллельных workflow с worktree.

Ваша задача: проанализировать этот проект и сгенерировать Coastfile (TOML-файл с именем "Coastfile" в корне проекта).

=== ДОКУМЕНТАЦИЯ ===

У Coast есть встроенная документация, доступная из CLI. Используйте её, чтобы понять полную схему Coastfile, стратегии томов, поведение assign и другие параметры конфигурации перед генерацией Coastfile.

Просмотреть дерево документации:

  coast docs

Это выводит полное дерево документации. Начните с чтения файлов README — они предоставляют индексы, которые помогут найти нужную документацию по каждой теме:

  coast docs --path README.md
  coast docs --path coastfiles/README.md
  coast docs --path concepts_and_terminology/README.md

Прочитать конкретный документ:

  coast docs --path coastfiles/PROJECT.md
  coast docs --path coastfiles/VOLUMES.md

Поиск по документации (семантический поиск — опишите естественным языком, что вы ищете):

  coast search-docs "how do volume strategies work"
  coast search-docs "shared postgres across instances"
  coast search-docs "secret injection from environment variables"

Используйте документацию, чтобы принимать обоснованные решения о конфигурации Coastfile для этого проекта. Раздел coastfiles/ подробно описывает каждую директиву Coastfile.

=== СХЕМА COASTFILE (краткая справка) ===

[coast] — Обязательно. Метаданные проекта.

  name        (string, required)  Идентификатор проекта, используемый в именовании контейнеров/томов.
  compose     (string, optional)  Путь к docker-compose.yml относительно Coastfile.
  runtime     (string, optional)  "dind" (по умолчанию), "sysbox" или "podman".
  root        (string, optional)  Переопределение корня проекта (относительный или абсолютный путь).

[coast.setup] — Опционально. Настройка самого DinD-контейнера.

  packages    (array of strings)  Пакеты Alpine для установки (например, ["nodejs", "npm", "git"]).
  run         (array of strings)  Произвольные команды для выполнения во время настройки.

[ports] — Обязательно (хотя бы один). Карта логического имени в номер порта.
  Эти порты пробрасываются на хост, когда coast развернут.

  Пример:
  [ports]
  web = 3000
  api = 8080
  postgres = 5432

[volumes.*] — Опционально. Конфигурация для каждого тома.

  strategy        "isolated" (по умолчанию) или "shared"
  service         Имя сервиса Compose, которому принадлежит этот том.
  mount           Путь монтирования внутри контейнера сервиса.
  snapshot_source (только isolated) Инициализация из существующего имени тома.

  Пример:
  [volumes.postgres_data]
  strategy = "isolated"
  service = "db"
  mount = "/var/lib/postgresql/data"

[secrets.*] — Опционально. Извлечение и внедрение секретов.

  extractor   "file", "env", "command" или "macos-keychain"
  inject      "env:VAR_NAME" или "file:/path/in/container"
  ttl         Опциональный срок действия (например, "1h", "30m").

  Параметры, зависящие от extractor:
    file:            path = "./path/to/secret"
    env:             var = "HOST_ENV_VAR"
    command:         run = "echo secret-value"
    macos-keychain:  item = "keychain-item-name"

  Пример:
  [secrets.db_password]
  extractor = "env"
  var = "DB_PASSWORD"
  inject = "env:DATABASE_PASSWORD"

[inject] — Опционально. Внедрение (инъекция) не-секретных файлов/переменных окружения с хоста.

  env     Массив имён переменных окружения хоста для проброса.
  files   Массив путей к файлам на хосте для монтирования.

  Пример:
  [inject]
  env = ["NODE_ENV", "DEBUG"]
  files = ["~/.ssh/id_ed25519", "~/.gitconfig"]

[shared_services.*] — Опционально. Сервисы на хостовом Docker daemon, общие для всех инстансов.

  image           Docker-образ.
  ports           Массив номеров портов.
  volumes         Массив монтирований томов.
  env             Inline-таблица переменных окружения.
  auto_create_db  (bool) Автоматически создавать БД для каждого инстанса.
  inject          Внедрить строку подключения в контейнеры coast.

  Пример:
  [shared_services.postgres]
  image = "postgres:16-alpine"
  ports = [5432]
  volumes = ["postgres_data:/var/lib/postgresql/data"]
  env = { POSTGRES_USER = "dev", POSTGRES_PASSWORD = "dev", POSTGRES_DB = "app" }
  auto_create_db = true
  inject = "env:DATABASE_URL"

[assign] — Опционально. Управляет тем, что происходит при переключении ветки (coast assign).

  default                 "none", "restart" или "rebuild"
  [assign.services]       Переопределения по сервисам.
  [assign.rebuild_triggers]  Глоб-паттерны файлов по сервисам, которые триггерят rebuild.

  Пример:
  [assign]
  default = "none"
  [assign.services]
  api = "restart"
  worker = "rebuild"
  [assign.rebuild_triggers]
  worker = ["Dockerfile", "package.json"]

[services.*] — Опционально. Сервисы как «голые» процессы (docker-compose не нужен).

  command   Команда shell для запуска.
  port      Номер порта.
  restart   "on-failure" или "always".

  Пример:
  [services.web]
  command = "node server.js"
  port = 3000
  restart = "on-failure"

=== ПРИМЕР: Минимальный (без compose) ===

[coast]
name = "my-app"
runtime = "dind"

[coast.setup]
packages = ["nodejs", "npm"]

[ports]
app = 3000

=== ПРИМЕР: С docker-compose ===

[coast]
name = "my-app"
compose = "./docker-compose.yml"
runtime = "dind"

[ports]
app = 3000
postgres = 5432
redis = 6379

[volumes.postgres_data]
strategy = "shared"
service = "db"
mount = "/var/lib/postgresql/data"

[volumes.redis_data]
strategy = "isolated"
service = "cache"
mount = "/data"

[assign]
default = "none"

[assign.services]
app = "rebuild"

=== ПРИМЕР: С секретами ===

[coast]
name = "my-app"
compose = "./docker-compose.yml"
runtime = "dind"

[ports]
app = 3000

[secrets.api_key]
extractor = "env"
var = "API_KEY"
inject = "env:API_KEY"

[secrets.ssh_key]
extractor = "file"
path = "~/.ssh/id_ed25519"
inject = "file:/run/secrets/ssh_key"

=== КЛЮЧЕВЫЕ КОМПРОМИССЫ ДЛЯ ОБСУЖДЕНИЯ С ПОЛЬЗОВАТЕЛЕМ ===

Перед генерацией Coastfile уточните у пользователя любые неоднозначные выборы конфигурации. Вот основные:

Стратегия базы данных и инфраструктуры — есть три варианта для сервисов вроде postgres и redis:
  - Isolated volumes (по умолчанию): каждый инстанс Coast получает собственную копию данных внутри своего DinD-контейнера. Инстансы не могут мешать друг другу. Лучший вариант, когда нужно состояние БД на ветку.
  - Shared volumes: все инстансы читают и пишут в один и тот же том внутри своих DinD-контейнеров. Экономит место на диске, но одновременная запись из нескольких инстансов может повредить данные.
  - Shared services: запускать базу данных на хостовом Docker daemon вместо запуска внутри каждого Coast. Все инстансы подключаются к одному общему серверу. Использует меньше всего памяти, поддерживает auto_create_db для БД на инстанс в рамках одного postgres, а данные сохраняются дольше, чем живёт инстанс (переживают удаление инстанса). Лучший вариант для больших команд или машин с ограниченной памятью.
  - Если в проекте есть база данных, спросите пользователя, какой подход он хочет. Объясните компромиссы — isolated самый безопасный, shared services наиболее эффективен по памяти.

Стратегия assign — что происходит при переключении Coast между worktree:
  - "none": ничего не делать (для сервисов вроде postgres/redis, которые не меняются между ветками).
  - "restart": перезапустить контейнер (для интерпретируемых сервисов, которым нужен только перезапуск процесса).
  - "rebuild": пересобрать Docker-образ и перезапустить (для сервисов, где смена ветки влияет на Dockerfile или зависимости сборки).
  - Если в проекте несколько сервисов, спросите, какие из них требуют rebuild, а какие — restart при переключении ветки.

=== ИНСТРУКЦИИ ===

1. Посмотрите на структуру этого проекта. Если есть docker-compose.yml, прочитайте его, чтобы определить сервисы, порты и тома.
2. Прочитайте релевантную документацию Coast (используйте `coast docs` и `coast search-docs`), чтобы понять стратегии томов, поведение assign и любые параметры конфигурации, применимые к стеку этого проекта.
3. Уточните у пользователя любые неоднозначные выборы конфигурации (см. компромиссы выше). Не гадайте — объясните варианты и дайте им выбрать.
4. Сгенерируйте Coastfile в корне проекта на основе анализа проекта и ввода пользователя.
5. Если в проекте нет docker-compose.yml, используйте [services.*] для определений «голых» процессов или [coast.setup] для установки зависимостей.
6. Запустите `coast build`. Если он завершится с ошибкой, проверьте ошибку и обратитесь к документации (`coast search-docs "<error description>"`) для устранения неполадок.
7. Запустите `coast run dev-1`. Если он завершится с ошибкой, проверьте ошибку и обратитесь к документации.
8. Запустите `coast ui`, чтобы открыть панель Coastguard (это для пользователя, когда вы закончите).
