# Coast Runtime

Этот проект использует Coasts — контейнеризированные runtime-среды для запуска сервисов, тестов
и других runtime-команд. Файловая система является общей между хостом и
контейнером, поэтому изменения файлов с любой стороны сразу видны обеим.

## Discovery

Перед первой runtime-командой в сессии выполните:

  coast lookup

Это выводит имя экземпляра, порты и примеры команд. Используйте имя экземпляра
из вывода для всех последующих команд.

## What runs where

Файловая система общая, поэтому используйте `coast exec` только для того, что требует
runtime контейнера (базы данных, сервисы, интеграционные тесты). Всё остальное
запускается напрямую на хосте.

Используйте `coast exec` для:
- Тестов, которым нужны работающие сервисы (юнит-тесты, интегрированные с сервисами или dbs, интеграционные тесты, API-тесты)
- Перезапуска сервисов или операций compose
- Всего, что обращается к базам данных, кешам или другим контейнерным сервисам

Запускайте напрямую на хосте:
- Линтинг, проверку типов, форматирование
- Git-операции
- Playwright и браузерные тесты
- Установку зависимостей на стороне хоста (npm install, pip install)
- Поиск по файлам, генерацию кода, статический анализ

Пример:

  coast exec <instance> -- sh -c "cd <dir> && npm test"    # нужна DB
  npm run lint                                              # хост подходит
  npx playwright test                                       # хост подходит

## Runtime feedback

  coast ps <instance>
  coast logs <instance> --service <service>
  coast logs <instance> --service <service> --tail 50

## Creating and assigning Coasts

Если `coast lookup` не возвращает совпадений, выполните `coast ls`, чтобы посмотреть, что существует.

Если для этого проекта уже запущен неназначенный Coast, предпочтительнее
назначить ему ваш worktree, а не создавать новый:

  coast assign <existing> -w <worktree>

Уже занятый Coast тоже можно переназначить с помощью `coast assign`, но сначала уточните
у пользователя, потому что это прервёт работу текущего слота.

Если ни один Coast не запущен, спросите пользователя перед созданием — Coasts могут
требовать много памяти:

  coast run <name> -w <worktree>

Проект должен быть собран до того, как можно будет создавать экземпляры. Если `coast run` завершается с ошибкой
из-за отсутствия сборки, сначала выполните `coast build`.

## Coastfile setup

Coastfile, сначала прочитайте документацию по Coastfile:

  coast docs --path coastfiles/README.md

## Rules

- Всегда запускайте `coast lookup` перед первой runtime-командой в сессии.
- Используйте `coast exec` только для того, что требует runtime контейнера.
- Запускайте линтинг, проверку типов, форматирование и git напрямую на хосте.
- Используйте `coast docs` или `coast search-docs`, прежде чем делать предположения о поведении Coast.
- Не запускайте сервисы напрямую на хосте, если проект предполагает использование Coast.

# Coasts

Используйте CLI Coast напрямую. Не добавляйте обёртки.

## Orient Yourself

Начните с изучения CLI и документации:

  coast                                             # посмотреть все доступные команды
  coast docs                                        # список всех страниц документации
  coast search-docs "your question"                 # семантический поиск

Если что-то в поведении Coast непонятно, читайте документацию, прежде чем делать предположения:

  coast docs --path concepts_and_terminology/RUN.md
  coast docs --path concepts_and_terminology/BUILDS.md
  coast docs --path concepts_and_terminology/ASSIGN.md
  coast docs --path concepts_and_terminology/PORTS.md
  coast docs --path coastfiles/README.md

## Quick Start

Направляйте запросы в один из этих режимов:

1. **Использовать Coast** — выполните `coast lookup`, затем используйте `coast exec`, `coast ps`
   или `coast logs` с подходящим экземпляром.
2. **Создать или назначить** — выполните `coast ls`, затем `coast run`, чтобы создать новый
   Coast, или `coast assign`, чтобы переназначить существующий.
3. **Открыть UI** — выполните `coast ui`.

## What Runs Where

Хост и Coast используют общую файловую систему. Используйте `coast exec` только для того,
что требует работающих внутри контейнера сервисов.

**Используйте `coast exec` для:**
- Интеграционных тестов, API-тестов и всего, чему нужны базы данных или сервисы
- Перезапуска сервисов, операций compose
- Команд, которые взаимодействуют с процессами, существующими только в контейнере

**Запускайте на хосте:**
- Линтинг (`eslint`, `rubocop`, `golangci-lint`)
- Проверку типов (`tsc --noEmit`, `go vet`)
- Форматирование (`prettier`, `gofmt`)
- Git-операции
- Playwright и браузерные тесты
- Статический анализ, генерацию кода
- Установку пакетов (`npm install`, `pip install`)

## Create and Assign

Когда `coast lookup` не возвращает совпадений:

1. Выполните `coast ls`, чтобы посмотреть доступные слоты.
2. Предпочитайте `coast run <name> -w <worktree>`, чтобы создать и назначить за один шаг.
3. Если сборки ещё нет, сначала выполните `coast build`.
4. После создания снова выполните `coast lookup` для подтверждения.

Когда вы хотите переключить существующий Coast на другой worktree:

  coast assign <name> -w <worktree>

Это также работает для уже назначенного или checked-out Coast, но сначала спросите пользователя,
прежде чем переназначать занятый слот.

## Coastfile Setup

Если проекту нужен новый или изменённый Coastfile, сначала прочитайте документацию:

  coast docs --path coastfiles/README.md

Документация по Coastfile охватывает настройку compose, порты, тома, секреты, общие
сервисы, bare-сервисы и наследование.

## Safety Rules

- Выполняйте `coast lookup` перед любыми действиями и снова после любого изменения топологии.
- Спрашивайте перед `coast assign`, `coast unassign` или `coast checkout`, если это
  нарушит работу существующего слота.
- Предпочитайте создание нового Coast повторному использованию checked-out или уже назначенного
  экземпляра, если только пользователь явно не хочет переназначить существующий слот.
- Используйте `coast docs` или `coast search-docs`, прежде чем делать предположения.
