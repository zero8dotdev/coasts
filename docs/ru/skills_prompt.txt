# Coast Runtime

Этот проект использует Coasts (containerized host) для изолированных сред разработки.
Ваши изменения в коде автоматически видны внутри запущенного Coast — файловая система
является общей между хостом и контейнером.

## Orientation

Перед выполнением любых runtime-команд определите, какой экземпляр Coast соответствует
вашей текущей рабочей директории:

```
coast lookup
```

Это выводит имя экземпляра, порты, URL и примеры команд. Используйте имя экземпляра
из вывода для всех последующих команд.

Если вам нужен более глубокий контекст о том, как работают Coasts, прочитайте эту документацию:

```
coast docs --path concepts_and_terminology/LOOKUP.md
coast docs --path concepts_and_terminology/FILESYSTEM.md
coast docs --path concepts_and_terminology/EXEC_AND_DOCKER.md
coast docs --path concepts_and_terminology/LOGS.md
```

## Running Commands

Используйте `coast exec`, чтобы выполнять команды внутри Coast. Оболочка запускается в корне
рабочей области (там, где находится Coastfile). Сначала перейдите в нужную директорию с помощью cd:

```
coast exec <instance> -- sh -c "cd <dir> && <command>"
```

Примеры:

```
coast exec dev-1 -- sh -c "cd src && npm test"
coast exec dev-1 -- sh -c "cd backend && go test ./..."
coast exec dev-1 -- sh -c "cd apps/web && npx playwright test"
```

## Runtime Feedback

Проверьте статус сервиса:

```
coast ps <instance>
```

Прочитайте логи сервиса:

```
coast logs <instance> --service <service>
coast logs <instance> --service <service> --tail 50
```

## Troubleshooting

Если вы сталкиваетесь с ошибками или незнакомым поведением, выполните поиск по документации Coast:

```
coast search-docs "error message or description"
```

Здесь используется семантический поиск — опишите проблему естественным языком, и он
найдет релевантную документацию.

## Worktree Awareness

Когда вы начинаете работать в worktree — независимо от того, создали ли вы его сами или это сделал
такой инструмент, как Codex, Conductor или T3 Code — проверьте, назначен ли уже
экземпляр Coast:

```
coast lookup
```

Если `coast lookup` находит экземпляр, используйте его для всех runtime-команд.

Если он не возвращает ни одного экземпляра, проверьте, что сейчас запущено:

```
coast ls
```

Затем спросите пользователя, какой вариант он предпочитает:

**Option 1 — Создать новый Coast и назначить этот worktree:**
```
coast run <new-name>
coast assign <new-name> -w <worktree>
```

**Option 2 — Переназначить существующий Coast этому worktree:**
```
coast assign <existing-name> -w <worktree>
```

**Option 3 — Полностью пропустить Coast:**
Продолжайте без runtime-среды. Вы можете редактировать файлы, но не можете запускать тесты,
сборки или сервисы внутри контейнера.

Значение `<worktree>` — это имя ветки (выполните `git branch --show-current`) или
идентификатор worktree, показанный в `coast ls`. Всегда спрашивайте пользователя перед созданием
или переназначением — не делайте этого автоматически.

## Rules

- Всегда запускайте `coast lookup` перед первой runtime-командой в сессии.
- Не запускайте сервисы напрямую на хосте. Используйте `coast exec` для всех runtime-задач.
- Изменения файлов на хосте мгновенно видны внутри Coast. Вам не нужно
  копировать файлы или пересобирать после редактирования.
- Если `coast lookup` не возвращает ни одного экземпляра, Coast может быть не запущен. Следуйте
  описанному выше процессу worktree awareness, чтобы решить это вместе с пользователем.
