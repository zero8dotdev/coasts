你正在将 Coasts 安装到这个项目中。Coast（容器化主机）是一个 CLI 工具，它使用 Docker-in-Docker 容器在一台机器上运行多个相互隔离的开发环境。每个环境都有自己的端口、卷和运行时——非常适合并行 worktree 工作流。

你的工作:分析这个项目并生成一个 Coastfile（项目根目录下名为 "Coastfile" 的 TOML 文件）。

=== 文档 ===

Coast 提供了可从 CLI 访问的内置文档。在生成 Coastfile 之前，使用这些文档来理解完整的 Coastfile schema、卷策略、assign 行为以及其他配置选项。

浏览文档树:

  coast docs

这会打印完整的文档树。从阅读 README 文件开始——它们提供索引，帮助你找到每个主题的正确文档:

  coast docs --path README.md
  coast docs --path coastfiles/README.md
  coast docs --path concepts_and_terminology/README.md

阅读一篇特定文档:

  coast docs --path coastfiles/PROJECT.md
  coast docs --path coastfiles/VOLUMES.md

搜索文档（语义搜索——用自然语言描述你要找的内容）:

  coast search-docs "how do volume strategies work"
  coast search-docs "shared postgres across instances"
  coast search-docs "secret injection from environment variables"

使用这些文档来对本项目的 Coastfile 配置做出明智决策。coastfiles/ 章节详细涵盖了每一个 Coastfile 指令。

=== COASTFILE SCHEMA（快速参考） ===

[coast] — 必需。项目元数据。

  name        (string, required)  用于容器/卷命名的项目标识符。
  compose     (string, optional)  相对于 Coastfile 的 docker-compose.yml 路径。
  runtime     (string, optional)  "dind"（默认）、"sysbox" 或 "podman"。
  root        (string, optional)  项目根目录覆盖（相对或绝对）。

[coast.setup] — 可选。自定义 DinD 容器本身。

  packages    (array of strings)  要安装的 Alpine 包（例如 ["nodejs", "npm", "git"]）。
  run         (array of strings)  在 setup 期间运行的任意命令。

[ports] — 必需（至少一个）。逻辑名称到端口号的映射。
  当 coast 被检出时，这些端口会转发到宿主机。

  示例:
  [ports]
  web = 3000
  api = 8080
  postgres = 5432

[volumes.*] — 可选。按卷的配置。

  strategy        "isolated"（默认）或 "shared"
  service         拥有该卷的 Compose 服务名。
  mount           服务容器内的挂载路径。
  snapshot_source (仅 isolated) 从现有卷名进行种子初始化。

  示例:
  [volumes.postgres_data]
  strategy = "isolated"
  service = "db"
  mount = "/var/lib/postgresql/data"

[secrets.*] — 可选。机密提取与注入。

  extractor   "file"、"env"、"command" 或 "macos-keychain"
  inject      "env:VAR_NAME" 或 "file:/path/in/container"
  ttl         可选的过期时间（例如 "1h"、"30m"）。

  提取器特定参数:
    file:            path = "./path/to/secret"
    env:             var = "HOST_ENV_VAR"
    command:         run = "echo secret-value"
    macos-keychain:  item = "keychain-item-name"

  示例:
  [secrets.db_password]
  extractor = "env"
  var = "DB_PASSWORD"
  inject = "env:DATABASE_PASSWORD"

[inject] — 可选。非机密的宿主机文件/环境变量注入。

  env     要转发的宿主机环境变量名数组。
  files   要挂载的宿主机文件路径数组。

  示例:
  [inject]
  env = ["NODE_ENV", "DEBUG"]
  files = ["~/.ssh/id_ed25519", "~/.gitconfig"]

[shared_services.*] — 可选。宿主机 Docker daemon 上跨实例共享的服务。

  image           Docker 镜像。
  ports           端口号数组。
  volumes         卷挂载数组。
  env             环境变量的内联表。
  auto_create_db  (bool) 自动创建每个实例一个数据库。
  inject          将连接字符串注入到 coast 容器中。

  示例:
  [shared_services.postgres]
  image = "postgres:16-alpine"
  ports = [5432]
  volumes = ["postgres_data:/var/lib/postgresql/data"]
  env = { POSTGRES_USER = "dev", POSTGRES_PASSWORD = "dev", POSTGRES_DB = "app" }
  auto_create_db = true
  inject = "env:DATABASE_URL"

[assign] — 可选。控制分支切换时发生什么（coast assign）。

  default                 "none"、"restart" 或 "rebuild"
  [assign.services]       针对每个服务的覆盖。
  [assign.rebuild_triggers]  针对每个服务触发 rebuild 的文件 glob。

  示例:
  [assign]
  default = "none"
  [assign.services]
  api = "restart"
  worker = "rebuild"
  [assign.rebuild_triggers]
  worker = ["Dockerfile", "package.json"]

[services.*] — 可选。裸进程服务（无需 docker-compose）。

  command   要运行的 shell 命令。
  port      端口号。
  restart   "on-failure" 或 "always"。

  示例:
  [services.web]
  command = "node server.js"
  port = 3000
  restart = "on-failure"

=== 示例:最小化（无 compose） ===

[coast]
name = "my-app"
runtime = "dind"

[coast.setup]
packages = ["nodejs", "npm"]

[ports]
app = 3000

=== 示例:使用 docker-compose ===

[coast]
name = "my-app"
compose = "./docker-compose.yml"
runtime = "dind"

[ports]
app = 3000
postgres = 5432
redis = 6379

[volumes.postgres_data]
strategy = "shared"
service = "db"
mount = "/var/lib/postgresql/data"

[volumes.redis_data]
strategy = "isolated"
service = "cache"
mount = "/data"

[assign]
default = "none"

[assign.services]
app = "rebuild"

=== 示例:使用 secrets ===

[coast]
name = "my-app"
compose = "./docker-compose.yml"
runtime = "dind"

[ports]
app = 3000

[secrets.api_key]
extractor = "env"
var = "API_KEY"
inject = "env:API_KEY"

[secrets.ssh_key]
extractor = "file"
path = "~/.ssh/id_ed25519"
inject = "file:/run/secrets/ssh_key"

=== 需要与用户讨论的关键权衡点 ===

在生成 Coastfile 之前，向用户询问任何不明确的配置选择。主要包括:

数据库与基础设施策略 — 针对 postgres、redis 等服务有三种选项:
  - 隔离卷（默认）:每个 Coast 实例在其 DinD 容器内都有自己的一份数据副本。实例之间互不影响。适合需要按分支保留数据库状态的场景。
  - 共享卷:所有实例在其 DinD 容器内读写同一个卷。节省磁盘空间，但多个实例并发写入可能会破坏数据。
  - 共享服务:在宿主机 Docker daemon 上运行数据库，而不是在每个 Coast 内部运行。所有实例连接到一个共享服务器。占用内存最少，支持 auto_create_db 在单个 postgres 上为每个实例创建数据库，并且数据在实例删除后仍然保留。适合大型团队或内存受限的机器。
  - 如果项目包含数据库，询问用户希望采用哪种方式。解释权衡点——isolated 最安全，shared services 最节省内存。

Assign 策略 — 在 worktree 之间切换 Coast 时会发生什么:
  - "none":不做任何事（适用于 postgres/redis 等在不同分支间不变化的服务）。
  - "restart":重启容器（适用于解释型服务，只需要重启进程）。
  - "rebuild":重建 Docker 镜像并重启（适用于分支切换会影响 Dockerfile 或构建依赖的服务）。
  - 如果项目有多个服务，询问哪些在分支切换时需要 rebuild，哪些只需要 restart。

=== 说明 ===

1. 查看这个项目的结构。如果有 docker-compose.yml，读取它以识别服务、端口和卷。
2. 阅读相关 Coast 文档（使用 `coast docs` 和 `coast search-docs`）以理解卷策略、assign 行为，以及适用于该项目技术栈的任何配置选项。
3. 向用户询问任何不明确的配置选择（见上面的权衡点）。不要猜测——解释选项并让用户决定。
4. 基于项目分析和用户输入，在项目根目录生成 Coastfile。
5. 如果项目没有 docker-compose.yml，使用 [services.*] 定义裸进程，或使用 [coast.setup] 安装依赖。
6. 运行 `coast build`。如果失败，检查错误并查阅文档（`coast search-docs "<error description>"`）进行排查。
7. 运行 `coast run dev-1`。如果失败，检查错误并查阅文档。
8. 运行 `coast ui` 打开 Coastguard 仪表盘（当你完成后，这是给用户使用的）。
